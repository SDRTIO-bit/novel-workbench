# 交接：检测器“绿色 / 人工特征”文本行为分析

**日期：** 2026-07-17  
**项目：** Novel Workbench  
**目的：** 交接“为什么局部段落会被外部检测器标为绿色（人工特征）”的现有证据、合理推断与实施边界。  
**重要前提：** 外部检测器的颜色和百分比不是作者身份的事实判定，也不是可逆推出的文法。本文只把它当作一组有噪声的外部反馈，不能把“刷绿”当成唯一质量目标。

---

## 1. 已观察到的事实

### 1.1 绿色不是固定句式造成的

用户提供的两组截图中，被标为绿色的并不是同一段、也不是同一种表面句法：

- 第一组是失重救援场景的一段连续动作：抓住、拉起、抛出、固定、倒计时、重力恢复；其中仍有直接心理句和短句计数。
- 第二组是同一章的另一版本：人物恐惧、身体被拉向边缘、主角用肩膀和缆绳固定自己；绿色区域的位置与第一组不同。
- 另有绿色片段来自维修记录 / 工单编号场景：人物看到具体编号，停顿、追问、回答，信息改变了下一步对话。

所以，不能得出“某个词、某个标点、短句、数字、动作描写 = 人工”的结论。相同主题换一种上下文后，颜色就会变化。

### 1.2 绿色片段共享的是局部叙事行为

虽然表面不同，绿色片段大多有这条可观察链：

```text
现场可见事实
→ 人物按有限信息作判断（可以错）
→ 立刻采取具体动作
→ 物理 / 社会后果改变下一步可做的事
```

例如不是“危险正在逼近，主角感到紧张”，而是“身体已被拉离地面；手边只剩护栏 / 缆绳；人物选择抓住或切断；该动作造成新的位置、时间或资源限制”。

这种链条不必宏大。一个工单编号让人物改口问人名，也属于有效的局部因果转折。

### 1.3 人工样本并不回避直白解释

用户提供的纯人工商业网文样本并非“纯展示文学”：

- 它会在冲突结束后直接说明主角为什么不后悔；
- 会使用口语化吐槽、粗粝判断和夸张反应；
- 也有不精细、甚至语病式的句子。

它仍然可读的原因不是完全没有解释，而是解释紧贴当前事件，确认人物态度或把读者推入下一件事；不是停下来填写“主角很正义 / 她很严谨”的人物档案。

结论：**不要把“少解释”误设成硬目标。应删的是没有新信息、没有视角态度、没有后续行动的解释。**

## 2. 目前最可信的“绿色倾向”

以下是工作假设，不是检测器规则。后续必须用真实生成样本 + 外部反馈继续验证。

| 倾向 | 绿色片段为什么可能受益 | 写作 / 系统含义 |
|---|---|---|
| 从未完成的动作切入 | 读者先看见人物正在解决什么，不先听设定介绍 | 开头应优先给 `entry_pressure`，不是天气、世界观或人物标签 |
| 受约束的具体动作 | 护栏、缆绳、工单、面板、倒计时都限制行动 | 细节应能改变动作；不是“眼神、呼吸、嘴角”填充 |
| 错误判断实际改变行动 | 人物把敲击当管道故障，因此切断电源 | 允许误判，但误判要有可见后果，旁白不要立即公布正确答案 |
| 因果不是解释链而是行动链 | 看到证据后问话、拉绳、切闸 | 技术 / 设定信息出现后要尽快变成选择、命令、操作或损失 |
| 不均匀但有压力的节奏 | 短句、长句、倒计时、对话的长度由现场决定 | 不要机械“长短句交替”；允许某些段很短，某些段只写一个动作 |
| 具体可见的章末事实 | 面板显示身份、工单时间戳、物件落下 | 钩子应是事实 / 选择 / 物件，不是“更大危机正在靠近” |
| 人物有带偏差的声音 | 吐槽、嘴硬、临时找补、答非所问 | 角色不必总是高效沟通或正确理解彼此 |

## 3. 当前高风险的“AI 感”行为

下列问题比某个禁用词更重要，应该由 Critic 诊断并由 Reviser 定点修复。

### 3.1 叙述者代替读者完成理解

常见形式：

- 动作或对白已经表达，下一句再翻译情绪；
- “这意味着……”“他终于明白……”“她不知道的是……”；
- 把人物的瞬间判断写成正确的世界解释。

修法：保留证据、当前人物的最小反应与下一步行动；删除“标准答案”。

### 3.2 角色标签在正文中重复执行

常见形式：

- “他是一个谨慎的维修员”“她一向严谨”；
- 每次出场都重复相同的傲娇 / 冷淡 / 吐槽模板；
- 为展示性格专门安排一个标志性微动作。

修法：资料层可以保留角色描述；正文只写此时此地的选择。角色特征应影响“更可能做什么”，不规定“每次都如何表演”。

### 3.3 线索传送带与公式化升级

常见形式：

```text
异常 A → 解释一点 A → 异常 B → 解释一点 B → 更大的异常 C
```

或者每段都按“发现—震惊—解释—钩子”完整收束。

修法：本章只让一个主要干扰改变局面；允许信息暂时无名，允许某些问题只造成行动限制，不立刻升级成世界观谜题。

### 3.4 把钩子后又写成总结

常见形式：

```text
屏幕出现异常结果。
他意识到事情不简单。
他的下班计划泡汤了。
```

第二、三句往往是在替读者收束，不是在新增戏剧事实。

正确处理不是把它改成“他只能看着”。这种反应句同样会稀释钩子。应保留钩子承载的可见事实为最后一个非空段，例如屏幕上的识别文字本身。

### 3.5 为“具体”而堆通用微动作

高风险词群不是绝对禁止，而是密度风险：

- 微微、轻轻、缓缓、莫名、几分、一丝；
- 皱眉、抿唇、指尖、嘴角、呼吸、目光；
- 反复“愣了一下”“心头一跳”。

只有当这些动作改变人物下一步选择、空间关系或读者判断时才应保留。

## 4. 已证伪或不应采用的方案

### 4.1 “禁止解释、只留白”

错误。此前的极端例子删掉全部心理与叙述态度后，文本变成分镜脚本：动作—对白—动作，反而没有人物温度。

应改为：允许带人物声音的解释；禁止无时机、无态度、重复已知内容的解释。

### 4.2 “整章一键降 AI / 润色”

错误。第二模型倾向把原稿洗成工整、对称、标准的网文腔，常引入“微微一怔、空气安静、说不清的情绪”。

应改为：Critic 最多指出 3—5 个已定位问题；用户选择问题；Reviser 只做对应操作；Judge 可以保留原文或局部回滚。

### 4.3 连续叠加十种润色提示词

错误。丰富细节、删冗、调节奏、模仿风格、精准用词会互相抵消，最后离原稿更远。

原则：**一个问题选择一个最小修订操作。**

### 4.4 根据检测器自动改写正文

错误。检测器会诱导模型过拟合表面特征，导致故意残句、硬塞口语、随机加错字，形成另一种模板感。

检测反馈只能记录为外部结果，供人工和 Critic 复盘；不能直接当 Writer 或 Reviser 的自动重写目标。

### 4.5 复制参考小说的情节、人物或句序

不可采用。参考的人工小说只能提供抽象节奏参考：冲突如何进入、信息如何变成行动、如何停在事实处。不得复制其具体人物、设定、桥段或连续表达。

## 5. 当前实现状态（截至 `a20fd96`）

### 已实现

1. Planner 可输出 `tempo_guardrails`：
   - `entry_pressure`
   - `dominant_disruption`
   - `allowed_viewpoint_misread`
   - `disclosure_cap`
   - `must_remain_unclassified`
   - `stop_after`
   - 可选 `final_line_must_include`

2. Writer 接收场景计划和节奏护栏；默认提示已避免旧式“四拍结构”要求。

3. Critic 有新增问题类型：
   - `narrator_character_label`
   - `clue_conveyor_belt`
   - `formulaic_escalation`
   - `premature_classification`
   - `closing_summary_hook`

4. Reviser 支持 `de_label`、`de_chain` 操作，并明确禁止在 `revised_text` 中保留 `[P001]` 段落编号。

5. Judge 只允许评估用户选中的问题。未知问题 ID 或遗漏所选 ID 会使 Judge 候选失效。

6. `final_line_must_include` 是确定性准入校验：
   - Writer / Reviser 的最后一个非空段若不含该短语，候选会标为 `TEMPO_FINAL_LINE_MISMATCH`；
   - 系统拒绝该候选，但**不会自动修改正文**。

7. Context 预览与真实 Judge 执行都附带同一份已选问题范围，Mock Provider 也遵守此范围。

### 已验证

- 本地完整测试：`202 passed`。
- 真实 Provider（DeepSeek 兼容调用）已跑通 Planner、Writer、Critic、Reviser、Judge。
- 真实 Writer 在带 `final_line_must_include: "在岗人员"` 的调用中，最终段保留了该短语，候选通过。

### 尚未验证 / 不应声称

- 不能声称任意小说都会获得检测器绿色或低 AI 百分比；
- 没有建立足够大的“生成版本—外部检测结果—人工选择”数据集；
- Reviser 对复杂结尾仍可能拒绝候选，需要界面提供重试、保留初稿、人工编辑；
- 没有证据表明某一个模型或温度设置稳定优于其他配置。

## 6. 给下一位 AI 的推荐工作顺序

### P0：先验证，不再堆提示词

1. 通过**正确的 API 请求结构**运行：`StageOverrideRequest` 字段必须位于请求 JSON 顶层，不能嵌套成 `stage_override`。
2. 每次实验保存：实际提示词版本、模型、温度、完整输入、原稿、Critic 选择项、修订稿、Judge 结果、人工最终选择。
3. 使用 3—5 个不同题材 / 场景，不要只用科幻维修开头。
4. 把外部检测结果人工录入 `detector_feedback`，只做对照，不触发自动改写。

### P1：检查真实产品路径

1. 确认前端发出的覆盖字段是顶层字段，而非嵌套 `stage_override`；此前命令行试验曾因错误嵌套造成“约束没传到模型”的假失败。
2. 在 UI 中显示 `final_line_must_include` 是否启用，以及候选被拒绝的具体原因。
3. 当 Reviser 候选因结尾标记失效时，允许用户：
   - 重试相同问题；
   - 换 Reviser 模型；
   - 保留原稿；
   - 手动编辑；
   - 不允许系统静默裁剪文本。

### P2：建立真实偏好数据，而不是“学习文风”神话

对每章记录：

```text
最终选原稿 / 修订稿 / Judge 合并稿 / 手动稿
哪些 Critic 问题被忽略
哪些修订被手工恢复
外部检测器标出的范围
作者对该范围的人工判断：真实问题 / 误报 / 不确定
```

累计至少 20—30 个可比较样本后，才考虑统计：

- 哪个修订操作的接受率最高；
- 哪个模型更容易删掉有效钩子；
- “线索传送带”“结尾总结”等问题是否真的与检测器反馈相关；
- 作者到底偏好哪些粗糙但有生命力的表达。

## 一句话交接结论

检测器看起来更容易给“人物在有限信息下处理具体麻烦、判断会造成可见后果、结尾停在可见事实”的段落以人工特征；它并不单纯偏爱短句、口语、少解释或词汇混乱。项目下一步应验证这种结构性假设，而不是把所有正文改成迎合检测器的表面文风。
